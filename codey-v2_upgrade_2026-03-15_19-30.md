### Step 1.1 — Text-to-Speech (Codey speaks responses) ✅
- [x] `core/voice.py` — VoiceManager class wraps `termux-tts-speak`

- [x] `/voice listen` — one-shot voice input passed directly to agent
- [x] Fallback to typed input if STT fails or returns empty
- **Requires:** Termux:API app installed

### Step 1.3 — Voice Mode Toggle ✅
- [x] `/voice on` / `/voice off` in the REPL
- [x] When on: every response is spoken (only final answers, not tool output)
- [x] Smart filtering: skips `<tool>` blocks, code blocks — speaks prose only
- [x] Voice state saved to `~/.config/codey-v2/voice_config.json` (persists sessions)
- NOTE: Voice preference not wired into LearningManager/CODEY.md — uses own config
  file instead. Simpler and more reliable than injecting into the preferences system.

## Phase 2 — Static Analysis & Code Review Tools ✅ COMPLETE (v2.5.2 — 2026-03-16)
*Zero new models. Uses existing linters. High value for code quality.*

### Step 2.1 — Linter Integration ✅
- [x] `core/linter.py` — LintIssue dataclass + ruff/flake8/mypy runners
- [x] Auto-lint runs after every successful `write_file`/`patch_file` on .py files
- [x] Results appended to tool result string → agent sees them and can fix
- [x] Priority order: env CODEY_LINTER → ruff → flake8 → ast syntax-only
- [x] `get_available_linters()` reports what's installed
- NOTE: pylint not run by default (too slow on Termux, huge output). User can set
  CODEY_LINTER=pylint to force it. ruff covers ~95% of pylint's checks faster.
- NOTE: "User configures linter in CODEY.md" deferred — using CODEY_LINTER env var
  instead. Simpler to implement correctly without CODEY.md parse complexity.

### Step 2.2 — Pre-Write Review Gate ✅
- [x] `check_syntax()` in `core/linter.py` uses ast.parse (free, no tools)
- [x] `execute_tool()` in `core/agent.py` checks syntax before any Python write
- [x] Returns [ERROR] with line/reason if syntax broken — file never touched
- [x] Agent auto-retries with the error message as context

### Step 2.3 — Code Review Command ✅
- [x] `/review <file>` in main.py — runs all available linters
- [x] Colored output: red=errors, yellow=warnings, green=clean
- [x] Prompts user "Ask Codey to explain and fix?" — runs agent with lint context
- NOTE: "Scores files on complexity/security" is a future enhancement — current
  implementation covers style + syntax + type errors. Security scoring needs
  dedicated tool (bandit). Noted for Phase 7/8.

## Phase 3 — Git Enhancements

## Phase 4 — Code Vector Database (RAG)
*The core long-term intelligence system. Connects to everything.*

### Step 4.1 — Code Ingestion Pipeline
- Every code block Codey writes, reads, or receives from peers gets captured
- Chunk by function/class boundaries (not arbitrary character limits)
- Store in SQLite with: source, language, task type, timestamp, raw code
- Tag with origin: `codey`, `claude`, `gemini`, `qwen`, `user`

### Step 4.2 — Embedding & Indexing
- Use existing `sentence-transformers` (already installed) to embed each chunk
- Store embeddings in the existing SQLite state DB
- Index by: language, task type, origin, quality score
- Deduplicate: don't store near-identical code twice

### Step 4.3 — Quality Scoring System
- **High score triggers:**
  - Code ran successfully via shell tool without error
  - User accepted without modification
  - Peer CLI (Claude/Gemini) validated or produced it
- **Low score triggers:**
  - Code caused a traceback
  - Codey had to retry the task
  - User manually edited heavily before accepting
- Scores update over time as code is reused and tested

### Step 4.4 — Retrieval-Augmented Generation (RAG)
- On each new task, embed the user's request
- Retrieve top 2-3 most relevant high-scoring code examples
- Inject as "Relevant examples from your history" block in system prompt
- Respect context window budget — examples only added if space allows
- Examples cited by ID so user knows where they came from

### Step 4.5 — Auto Fine-Tuning Feed
- High-quality exchanges (quality score above threshold) automatically flagged
- Tagged with metadata: task type, language, peer-validated Y/N
- Feeds directly into the existing `--finetune` export pipeline
- Over time: Codey builds its own personalized training dataset passively
- User runs `codey2 --finetune` when ready and the dataset is already curated

## Phase 5 — Vision Model (Image Understanding)
*One model at a time. Swap in, describe, swap out. Feeds text to coding model.*

### Step 5.1 — Vision Model Setup
- Download Qwen2-VL-2B (GGUF) — approximately 1.5GB
- Download matching multimodal projection file (`mmproj`)
- Store alongside existing models in `~/models/`
- Add as a third model slot in `loader_v2.py` (swap architecture already exists)

### Step 5.2 — Image Detection & Routing
- Detect when a message contains an image file path (`.png`, `.jpg`, `.webp`, `.pdf`)
- Detect when user pastes a file path or types "look at this image / screenshot"
- Auto-trigger vision model swap when image detected
- Show clear indicator: "Loading vision model to analyze image..."

### Step 5.3 — Vision → Text → Coding Pipeline
- Vision model loads (coding model pauses via SIGSTOP — already implemented)
- Vision model analyzes image, produces detailed text description
- Vision model unloads, coding model resumes
- Text description injected into conversation as context
- Coding model continues with full image understanding as text

### Step 5.4 — Use Cases Enabled
- Screenshot of an error → Codey reads and fixes it
- UI mockup / wireframe → Codey generates the code for it
- Diagram or flowchart → Codey explains or implements it
- Photo of handwritten code or notes → Codey transcribes and works with it

## Phase 6 — External API Tool
*Extends what the shell tool already does. Needs a safety layer.*

### Step 6.1 — HTTP Request Tool
- Add `http_request` as a named agent tool alongside existing tools
- Supports GET, POST, PUT, DELETE with headers and body
- Response parsed and summarized before injecting into context
- Large responses truncated intelligently (first N lines + summary)

### Step 6.2 — Safety Layer (Allowlist)
- User defines allowed domains in CODEY.md or config
- Agent cannot call URLs outside the allowlist
- No credentials stored in conversation history (redacted like existing session redaction)
- Confirmation prompt for any POST/PUT/DELETE request

### Step 6.3 — Built-In Useful Integrations
- GitHub API — read issues, PRs, repo info directly in chat
- PyPI — check package versions, dependencies
- Documentation fetching — pull relevant docs for libraries being used
- All optional, user-configured, nothing automatic

## Phase 7 — NLP & Routing Improvements
*No new models. Smarter use of the existing model.*

### Step 7.1 — Better Intent Detection
- Expand task type detection beyond current keyword matching
- Use lightweight classifier (trained on conversation history) to detect intent
- Reduce false positives where tool use triggers on conversational messages
- Better detection of multi-step compound requests

### Step 7.2 — Conversation Memory Compression
- Smarter summarization that preserves code decisions and key facts
- Tag important turns (file written, decision made, error fixed) as "anchors"
- Anchor turns never get summarized away — always stay in context
- Reduces context pressure while keeping the important thread

### Step 7.3 — Proactive Suggestions
- After completing a task, Codey notices related improvements it could make
- Surfaces them as optional suggestions, not automatic actions
- Example: "I noticed you have no tests for this function. Want me to write some?"
- Driven by the linter output and code review data from Phase 2

## Phase 8 — Debugging Tools (pdb Integration)
*Builds on Phase 2 linter work. Closes the fix loop completely.*

### Step 8.1 — Auto Post-Mortem Debugging
- When `--fix` mode encounters a traceback, automatically run pdb post-mortem
- Capture local variable state at crash point
- Feed crash state + locals to agent as structured context
- Agent proposes fix with full knowledge of program state at failure

### Step 8.2 — Interactive Debug Session
- `/debug <file>` command launches file under pdb via PTY (same system as peer CLIs)
- User can step through code manually
- Codey watches the pdb output and offers suggestions at each step
- Ctrl+B returns to Codey with full debug transcript as context

### Step 8.3 — Test-Driven Debug Loop
- Combine with existing TDD mode (`--tdd`)
- Failing test → pdb post-mortem → agent fix → re-run test → repeat
- Fully automated loop with user confirmation gates
- Stops when all tests pass or max retries hit (escalates to peer CLI)

## Suggested Implementation Order & Rationale

| Phase | Why This Order |
|-------|---------------|
| **1 — Voice** | Fastest win, no infrastructure needed, immediately changes how you interact with Codey |
| **2 — Code Review** | No new models, high daily value, lays groundwork for quality scoring in Phase 4 |
| **3 — Git** | Foundation exists, extends something already working, low risk |
| **4 — Vector DB / RAG** | Needs Phase 2 quality signals to score code properly. Once running, feeds every later phase |
| **5 — Vision** | Needs stable swap infrastructure (already solid). RAG (Phase 4) can store vision-derived context |
| **6 — External API** | Needs stable agent loop and safety patterns established by earlier phases |
| **7 — NLP** | Benefits from having real conversation data (Phase 4 DB) to improve on |
| **8 — Debugging** | Most complex integration. Needs linters (Phase 2), stable loop (all prior phases) |

## Implementation Notes & Findings (added during upgrade work)

### Phase 1 (Voice) — 2026-03-16
- `termux-speech-to-text` output format varies by Termux:API version — handled both JSON
  and plain text output in `VoiceManager.listen()`.
- Voice state intentionally stored in `~/.config/codey-v2/voice_config.json` rather than
  learning system / CODEY.md — cleaner separation of concerns.
- TTS length capped at 1200 chars to avoid multi-minute speech runs on long responses.

### Bug Fix Detour (v2.5.3) — 2026-03-16
**Agent loop after simple writes (e.g. hello.py)**
- Root cause 1: prompt uses plain `User:` / `Assistant:` labels, model hallucinated
  the next user turn. `\nUser:` was NOT in stop sequences → model kept generating.
  Fix: added `"\nUser:"`, `"\nHuman:"`, `"\nA:"` to MODEL_CONFIG["stop"] AND to
  HALLUCINATION_MARKERS post-processing.
- Root cause 2: auto-lint (Phase 2) injected `F401 'sys' imported but unused` warning
  into agent tool result → agent saw outstanding work, kept looping trying to fix it.
  Fix: only inject linter ERRORS into agent context. Warnings go to user terminal only.

**extra_stop tokens (e.g. `</tool>`) were silently ignored**
- `_infer_http()` in `inference_v2.py` built a combined `stop` list with `extra_stop`
  but called `server.infer(prompt, max_tokens=...)` WITHOUT passing the stop list.
  `LlamaServer.infer()` then did its own `MODEL_CONFIG.get("stop", [])` lookup,
  losing all extra_stop entries entirely.
  Fix: added `stop: list = None` param to `LlamaServer.infer()` and updated
  `_infer_http` to pass the combined stop list through.

**CPU showing 0.0% always**
- Root cause: `_seed_cpu_proc()` in `__init__` and first `_loop_once()` in `start()`
  run microseconds apart → d_total ≈ 0 → 0.0%. Background thread subsequently
  also gets small deltas if inference activity is brief.
  Fix: added self-contained 250ms mini-sample in `_read_cpu_proc()` when d_total < 20.
  Also fixed psutil fallback: only trust psutil when it returns > 0.0 (first-call
  psutil.cpu_percent(interval=None) always returns 0.0).

### Phase 2 (Linter) — 2026-03-16
- pylint excluded from auto-lint (too slow on Termux for per-write runs) — ruff covers
  ~95% of pylint's checks in a fraction of the time. pylint available via CODEY_LINTER.
- Pre-write syntax gate uses Python's built-in `ast.parse` — works with zero external
  tools installed.
- Auto-lint result appended to tool result string so agent naturally sees it on the next
  inference step — no extra message injection needed.
- Future: Phase 4 (RAG) quality scoring should use linter pass/fail as one of its signals
  (clean code → higher quality score for vector DB storage).
- Future: `/review` could add `bandit` for security scanning (TODO for Phase 6/7).
