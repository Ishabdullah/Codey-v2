# Codey-v2 System Audit & Code Review
**Date:** Monday, March 9, 2026
**Status:** Completed

This report provides a comprehensive audit of the Codey-v2 codebase, focusing on security, stability, performance, and architectural integrity.

---

## Executive Summary
The system is a sophisticated local AI assistant designed for the Termux (Android) environment. It employs a multi-model routing strategy and an orchestrator for complex task decomposition. However, several **critical security vulnerabilities** and **major architectural bottlenecks** were identified that could lead to system compromise, instability, or poor user experience on resource-constrained devices.

---

## 1. Critical Issues (Action Required Immediately)

### 1.1. Arbitrary Shell Execution & Weak Sanitization
- **File Path:** `tools/shell_tools.py`
- **Issue:** The `shell` tool provides unrestricted access to the Termux environment. While a `DANGEROUS_COMMANDS` list exists, it is easily bypassed. The `--yolo` flag removes all user confirmation, allowing a model to execute potentially destructive commands without oversight.
- **Code Evidence:**
  ```python
  def shell(command: str, yolo: bool = False, timeout: int = 30) -> str:
      # ...
      if should_confirm and not yolo:
          if not ask_confirm(f"Run shell command: `{command}`?"):
              return "[CANCELLED] User declined to run command."
      # ...
      result = subprocess.run(command, shell=True, ...)
  ```
- **Reasoning:** A "jailbroken" or confused LLM could execute `rm -rf /` or exfiltrate private data from the device. The `is_dangerous` check only looks for the primary command name and a few patterns, which is insufficient against sophisticated evasion.

### 1.2. Self-Modification Vulnerability
- **File Path:** `core/filesystem.py`
- **Issue:** The filesystem abstraction explicitly allows the agent to write to the application's own source code directory (`CODE_DIR`), even if it is outside the defined workspace.
- **Code Evidence:**
  ```python
  def _validate_path(self, path: Union[str, Path]) -> Path:
      # ...
      try:
          path.relative_to(self.workspace)
      except ValueError:
          # Allow access to CODE_DIR for self-modification
          try:
              path.relative_to(CODE_DIR)
          except ValueError:
              raise FilesystemAccessError(...)
  ```
- **Reasoning:** This is a severe security risk. A model could overwrite its own logic to remove security checks, install persistent backdoors, or create infinite loops that crash the system.

## 2. High Priority Issues (Fix Recommended)

### 2.1. Performance Bottleneck: Model Hot-Swapping
- **File Path:** `core/loader_v2.py`
- **Issue:** To switch between the 7B (primary) and 1.5B (secondary) models, the system kills the active `llama-server` process and starts a new one.
- **Code Evidence:**
  ```python
  def _load_model(self, model_type: ModelType, model_path: Path) -> bool:
      # ...
      if self._loaded_model != model_type:
          self.unload() # Calls self._server.stop() which kills the process group
      self._server = LlamaServer(model_path, model_type)
      if not self._server.start(): ...
  ```
- **Reasoning:** On Android/Termux, loading a 7B model can take 30-60 seconds and consumes significant CPU/RAM. Frequent swapping during orchestrated tasks makes the system feel unresponsive and drains battery.

### 2.2. Fragile JSON Parsing & "Fixing"
- **File Path:** `core/agent.py`
- **Issue:** `extract_json` uses manual character-level depth tracking and aggressive regex to "fix" malformed LLM output (e.g., trailing commas, missing braces).
- **Code Evidence:**
  ```python
  cleaned = re.sub(r',\s*([}\]])', r'\1', candidate)
  # ...
  for key in ["name", "path", "content", ...]:
      m = re.search('"' + key + '"\\s*:\\s*"((?:[^"\\\\]|\\\\.)*)"', candidate, re.DOTALL)
  ```
- **Reasoning:** Manual JSON patching is error-prone. The trailing comma regex could accidentally modify valid strings containing that pattern. The fallback regex for key extraction is fragile and may fail on escaped characters or multi-line content.

---

## 3. Medium Priority Issues (Technical Debt & Reliability)

### 3.1. Brittle Hallucination Detection
- **File Path:** `core/agent.py`
- **Issue:** `is_hallucination` relies on simple string matching (keywords and phrases) to determine if a model is lying about performing an action.
- **Code Evidence:**
  ```python
  false_file = needs_file and not file_done and any(p in resp_lower for p in [
      "has been created", "was created", "have created", ...
  ])
  ```
- **Reasoning:** This leads to many false positives (where the model describes what it *will* do) and false negatives (where the model uses a phrase not in the list).

### 3.2. Heuristic-Based Orchestration
- **File Path:** `core/orchestrator.py`
- **Issue:** The decision to trigger a multi-step "plan" is based on simple keyword counting (`COMPLEX_SIGNALS`) and message length.
- **Reasoning:** Conversational questions like "How do I build a class?" often contain enough keywords to trigger the orchestrator, leading to unnecessary planning steps for simple Q&A.

### 3.3. Context Window Management
- **File Path:** `core/agent.py` / `core/context.py`
- **Issue:** While history is trimmed, the `build_file_context_block` injects full file contents into the system prompt based on user queries.
- **Reasoning:** There is no strict limit or priority-based selection for these files, meaning a large project could easily blow the 8k-32k context window of local models, causing the agent to lose its "instructions."

---

## 4. Low Priority & Optional (Code Quality)

### 4.1. Global State & Singletons
- **Files:** `core/filesystem.py`, `core/loader_v2.py`
- **Issue:** Use of global instances (`_filesystem`, `_loader`) makes the codebase difficult to unit test and prevents future support for concurrent agent sessions.

### 4.2. UI/Logic Coupling
- **Files:** Throughout `core/`
- **Issue:** Core logic files frequently import from `utils.logger` and `core.display` to print formatted output.
- **Reasoning:** This makes it difficult to use the core logic as a library (e.g., for a GUI or web-based frontend) without also inheriting the CLI-specific display logic.

## 5. Recommended Fixes & Regression Prevention Strategy

To ensure long-term stability and security, the following strategic fixes and automated validation measures are recommended.

### 5.1. Securing Shell Execution (Fix for 1.1)
- **Strategic Fix:** Shift from a "Deny-List" to an **"Allow-List"** of safe commands (e.g., `ls`, `git status`, `npm test`). For any command outside this list, enforce strict user confirmation regardless of the `yolo` flag. Use a formal argument parser to prevent sub-shell injection (e.g., `;`, `&&`, `|`).
- **Regression Prevention:** Implement a `tests/security/test_shell_sandbox.py` suite. This suite should include "Red Team" test cases that attempt to execute forbidden commands (e.g., `rm -rf /`, `curl`, `env`) and verify that they are blocked or correctly intercepted for confirmation.

### 5.2. Enforcing Filesystem Boundaries (Fix for 1.2)
- **Strategic Fix:** Completely remove the `CODE_DIR` exception in `Filesystem._validate_path`. The agent should only ever operate within the user-defined `WORKSPACE_ROOT`. If self-modification is a desired feature, it should be a separate, highly restricted tool with its own explicit opt-in.
- **Regression Prevention:** Add a unit test in `tests/test_filesystem.py` that specifically attempts a `write` operation to the `core/` directory and asserts that a `FilesystemAccessError` is raised. This test must run in every CI cycle.

### 5.3. Optimized Model Management (Fix for 2.1)
- **Strategic Fix:** Implement a **"Warm Process Pool"** or use a single `llama-server` instance with multiple loaded slots (if the hardware supports it). Alternatively, implement a LRU (Least Recently Used) cache for model processes so that switching back to a primary model doesn't require a full reload if it was used recently.
- **Regression Prevention:** Create an automated performance benchmark script that measures "Time to First Token" after a model swap. Set a maximum threshold for this metric to ensure future changes don't degrade the user experience further.

### 5.4. Robust JSON Extraction (Fix for 2.2)
- **Strategic Fix:** Replace the manual regex and character-looping logic with a specialized **"Relaxed JSON" parser** (e.g., `dirtyjson` or `json5`). These libraries are specifically designed to handle common LLM artifacts like trailing commas, unquoted keys, and missing closing braces more reliably than custom regex.
- **Regression Prevention:** Maintain a `tests/fixtures/malformed_tool_calls.json` file containing a collection of actual problematic outputs from models. Create a test that iterates through these fixtures and ensures the parser extracts the correct `name` and `args` every time.

### 5.5. Heuristic Improvement & Test-Driven Orchestration (Fix for 3.1 & 3.2)
- **Strategic Fix:** Move orchestration and hallucination logic into a dedicated **"Validation Layer"** that uses the secondary (fast) model to verify the primary model's claims. A 1.5B model can quickly scan a response and flag if it claims a file was created without a corresponding tool call.
- **Regression Prevention:** Build a "Golden Set" of user requests and expected task plans. Run a regression test that verifies the orchestrator produces the expected number of steps for these requests, preventing "plan bloat" or missed steps in future updates.

### 5.6. Architectural Decoupling (Fix for 4.1 & 4.2)
- **Strategic Fix:** Refactor the system to use **Dependency Injection**. Pass the `Filesystem` and `ModelLoader` instances to the `Agent` rather than relying on global singletons. Abstract the UI into an `OutputProvider` interface that the core logic interacts with.
- **Regression Prevention:** This enables the creation of a full mock-based test suite where the agent's logic can be tested without actually touching the disk or starting a `llama-server`, leading to faster and more reliable testing.

---

## Conclusion
Codey-v2 is a powerful tool but requires significant hardening. The **Critical** security issues regarding arbitrary shell execution and self-modification should be addressed immediately by implementing a stricter sandbox and removing the `CODE_DIR` bypass. Improving model management and JSON parsing will significantly enhance the system's reliability and performance.

### 6.1 Critical Issues Re-Evaluation

#### 1.1 Arbitrary Shell Execution & Weak Sanitization

**Audit Recommendation:** Shift to allow-list, enforce strict confirmation regardless of `--yolo` flag, use argument parser to prevent injection.

| Sub-Recommendation | Decision | Rationale |
|--------------------|----------|-----------|
| Allow-list for daemon mode | ✅ IMPLEMENT | Appropriate for unattended background operation |
| Remove `--yolo` flag entirely | ❌ DO NOT IMPLEMENT | Violates user sovereignty design; `--yolo` is explicit user opt-in |
| Argument parser validation | ✅ IMPLEMENT | Prevents sub-shell injection (`;`, `&&`, `|`), low cost, high benefit |

**Design Context:**
Codey-v2 is designed as a **local, user-controlled tool** where the user accepts risks in exchange for capability. The `--yolo` flag is explicitly documented in `CODEY.md` and `codey-v2.md` as a user choice. Removing it would violate the core design principle of user sovereignty over their own device.

**Corrected Implementation Plan:**

```python
def _validate_command_structure(command: str) -> bool:

# core/task_executor.py - Keep allow-list for daemon mode
DAEMON_ALLOWED_PREFIXES = (

    "cd ", "pwd", "which", "env", "printenv",
)
```

#### 1.2 Self-Modification Vulnerability

**Audit Recommendation:** Completely remove `CODE_DIR` exception in `Filesystem._validate_path()`.

| Sub-Recommendation | Decision | Rationale |
|--------------------|----------|-----------|
| Remove `CODE_DIR` bypass entirely | ❌ DO NOT IMPLEMENT | Breaks core self-modification feature |
| Add explicit self-modification opt-in | ✅ IMPLEMENT | User consent for self-mod capability |
| Enforce checkpoint before core writes | ✅ IMPLEMENT | Safety net for core file modifications |

**Design Context:**
This is the **most critical misalignment** between the audit and Codey-v2's design. From `codey-v2.md`:

> **Feature 5: Self-Modification with Checkpointing**
> "Remove `PROTECTED_FILES` block. Agent can modify any file, including its own source. Before modifying core files, agent creates a checkpoint snapshot for rollback."

The audit treats self-modification as a **security vulnerability**, but it is an **intentional feature** for Codey-v2's evolution capability.

**Corrected Implementation Plan:**
```python
# core/filesystem.py::_validate_path()
def _validate_path(self, path: Union[str, Path]) -> Path:
    # ... existing validation ...
    
    # Allow CODE_DIR access ONLY if self-modification is explicitly enabled
    try:
        path.relative_to(CODE_DIR)
        if not self.allow_self_modification:
            raise FilesystemAccessError(
                "Self-modification disabled. Enable with --allow-self-mod or set ALLOW_SELF_MOD=true"
            )
        # Enforce checkpoint before core file writes
        self._require_checkpoint(path)
        return path
    except ValueError:
        pass
    
    raise FilesystemAccessError(...)
```

### 6.2 High Priority Issues Re-Evaluation

#### 2.1 Performance Bottleneck: Model Hot-Swapping

**Audit Recommendation:** Implement "Warm Process Pool" or multiple loaded slots.

| Sub-Recommendation | Decision | Rationale |
|--------------------|----------|-----------|
| LRU cache for model processes | ✅ IMPLEMENT | Quick restart without full reload |
| Multiple loaded slots | ❌ DO NOT IMPLEMENT | Exceeds 12GB RAM budget on target hardware |

**Design Context:**
From `codey-v2.md`:

> "Running both at once is impossible on 12GB RAM."

The target device (S24 Ultra) has 12GB RAM. Both models loaded simultaneously would consume ~10GB vs. ~5GB for single model, defeating the thermal/memory efficiency goal.

**Corrected Implementation Plan:**
```python
# core/loader_v2.py::ModelLoader
class ModelLoader:
    def __init__(self):
        self._stopped_servers: dict = {}  # Keep stopped servers for quick restart
    
    def unload(self):
        """Don't kill server, just stop it (keep in memory for quick restart)."""
        if self._server:
            self._stopped_servers[self._loaded_model] = self._server
            self._server.process.send_signal(signal.SIGSTOP)  # Pause, don't kill
```

#### 2.2 Fragile JSON Parsing & "Fixing"

**Audit Recommendation:** Replace with `dirtyjson` or `json5` library.

| Sub-Recommendation | Decision | Rationale |
|--------------------|----------|-----------|
| Use `dirtyjson`/`json5` | ❌ DO NOT IMPLEMENT | Dependency/compilation issues on Termux |
| Improve current parser | ✅ IMPLEMENT | Handle escaped characters, multi-line strings |
| Add test fixtures | ✅ IMPLEMENT | Regression prevention for malformed output |

**Design Context:**
Recommending `dirtyjson` or `json5` ignores the **Termux/Android deployment constraint**. These libraries may not have prebuilt wheels for Android, requiring compilation (C extensions), which increases deployment complexity.

**Corrected Implementation Plan:**
```python
# core/agent.py::extract_json()
def extract_json(raw):
    # ... existing logic ...
    
    # Add handling for escaped characters in strings
    for key in ["name", "path", "content", "command", "pattern", "old_str", "new_str"]:
        # Try quoted value first
        m = re.search(rf'"{key}"\s*:\s*"((?:[^"\\]|\\.)*)"', candidate, re.DOTALL)
        if m:
            result[key] = m.group(1).replace('\\n', '\n').replace('\\"', '"')
            continue
        # Try unquoted value (for paths, commands)
        m = re.search(rf'"{key}"\s*:\s*([^,}}\n]+)', candidate)
        if m:
            result[key] = m.group(1).strip().strip('"')
```

```python
MAX_FILE_CONTEXT_TOKENS = 4000  # Leave room for system prompt + history

def build_file_context_block(user_message: str, max_tokens: int = MAX_FILE_CONTEXT_TOKENS):
    from core.memory import memory as _mem
    from core.tokens import estimate_tokens
    
    files = _mem.get_files_by_relevance(user_message)
    
    context_parts = []
    current_tokens = 0
    
    for file_info in files:
        file_content = f"## {file_info.path}\n{file_info.content}"
        file_tokens = estimate_tokens(file_content)
        
        if current_tokens + file_tokens > max_tokens:
            break
        
        context_parts.append(file_content)
        current_tokens += file_tokens
    
    return "\n".join(context_parts)
```
